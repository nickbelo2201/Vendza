## Codebase Patterns

- **storeId nunca do body:** Resolver sempre por `STORE_SLUG` env (storefront) ou `partnerContext` (partner)
- **Envelope HTTP:** Todas as respostas usam `ok(data, { stub: false })` de `lib/http.js`
- **ESM imports locais:** Usar extensão `.js` mesmo em arquivos `.ts`
- **Prisma client:** Importar de `@vendza/database` (não do package diretamente)
- **OrderEvent:** append-only — nunca UPDATE ou DELETE
- **Server Component → CSS hover:** Nunca passar event handlers via props de Server para Client Component. Usar CSS `:hover` classes.
- **web-partner prefixo CSS:** `.wp-*`
- **web-client prefixo CSS:** `.wc-*`
- **Carrinho localStorage key:** `vendza_carrinho`
- **Age gate localStorage key:** `vendza_age_verified`
- **Typecheck:** `corepack pnpm typecheck` do diretório `app/`
- **Socket rooms:** `store:{storeId}` para partner, `order:{publicId}` para cliente
- **publicId format:** `PED-NNNN` (4 dígitos zero-padded, sequencial por store)

- **Learnings:**
  - Prisma v7 gerado no projeto NÃO exporta `Prisma.ProductWhereInput` pelo `@vendza/database` — tipos de where são do client gerado mas não re-exportados via namespace `Prisma`. Solução: usar spread de objetos condicionais direto no `findMany`.
  - O tipo de retorno do `prisma.model.findMany()` é `any[]` porque `PrismaClient` está declarado como `any` no `.d.ts` gerado. Para usar `.map()` sem erro de implicit any, usar `type Row = (typeof results)[number]` e anotar o parâmetro.
  - `reply.code(503).send(...)` com schema `response: { 200: ... }` causa erro TS em Fastify v5 (status 503 não é assignable to 200). Solução: usar `throw new Error(...)` para casos de configuração ausente.

## 2026-04-07T00:00:00Z — Análise de Board + Modelo de Negócio Revisado

### Contexto
O fundador revisou o posicionamento do produto. O Vendza não é commodity SaaS — é um serviço gerenciado + software + automação entregue de forma consultiva.

### Documentação atualizada
- `docs/01-visao-estrategica.md` — reescrito com posicionamento premium e argumento de ROI
- `docs/04-matriz-de-funcionalidades.md` — Mercado Pago marcado como 🔴 P0 CRÍTICO; status de cada feature atualizado

- `docs/17-modelo-de-negocio.md` — **NOVO** — modelo completo: tiers, precificação, GTM, unit economics, fluxo de venda, programa de indicação

### Novo Modelo de Negócio

| Tier | Preço/mês | Inclui |
|------|-----------|--------|
| Plataforma | R$ 500 | Canal de vendas + painel + CRM básico |
| Plataforma + Automação WhatsApp | R$ 1.500 | Tudo + WhatsApp automação 24h |
| Premium (com IA) | R$ 2.500–3.000 | Tudo + IA no atendimento + insights preditivos |

**Meta de receita:** R$ 5.000/mês de MRR até março de 2027 = 4 clientes no tier R$ 1.500.

### Argumento de ROI (versão correta)
Não usar substituição total — usar migração gradual:
> "Se migrar 40% dos pedidos do Zé pro canal próprio, você já empata com o Vendza. Break-even no tier R$ 1.500 com 40% migrado de uma adega que fatura R$ 15k pelo Zé."

### Decisões ratificadas pelo Board
- Ticket R$ 1.500 **defensável** para adegas com faturamento > R$ 30k/mês
- Meta 4 clientes em 11 meses é **conservadora** (funil precisa ~100 contatos frios)
- Contrato 6 meses + multa 2 meses **legalmente sólido** no Brasil
- Parceria com distribuidoras: **postergar** — validar B2C primeiro

### Correções no modelo
- Programa de indicação: trocar "1 mês grátis" por **R$ 300–500 Pix por indicação fechada**
- Não criar tier abaixo de R$ 500 — dilui posicionamento premium
- Considerar tier intermediário R$ 800–900 para adegas menores no futuro
